Build Your Prototype
with Claude Code

Tutorial ~20 minute read · Phase 1 + Phase 2 walkthrough

You've done the hard thinking in found-with-claude: mapped the opportunity, talked to customers, stress-tested with synthetic panels, and defined your product strategy. Step 4 ended with a product brief. Step 5 is where you build the thing — and ship it fast.

The order here is deliberate: prototype → ship → harden. Define what the agent should do, get it running in production quickly, then use evals to raise quality on the already-running agent. Don't wait for perfection before shipping. Real inputs reveal failure modes that no test suite anticipates.

This guide walks through prototype-with-claude end to end — from your first skill run to a production-ready Python agent. Every step is a real command you can run.

What you're actually building

The output of this step is a running production agent — plus an eval harness to harden it over time.

The agent is defined as a Skill: a plain-text file (SKILL.md) that tells Claude what to do, step by step. Claude Code runs it immediately — no deployment, no infrastructure. One command exports it to production Python. Then evals measure and improve quality on the already-running agent, with no redeployment required.

Setup — 2 minutes

You need Claude Code installed. That's the only requirement for Phase 1.

git clone https://github.com/sinned/prototype-with-claude
cd prototype-with-claude
claude

Claude Code reads CLAUDE.md automatically when it opens — this file tells Claude how to help you design skills, debug them, and generate SDK code. You don't need to set anything up; the instructions are already there.

No API key needed for Phase 1. Claude Code uses your Claude subscription. You only need an ANTHROPIC_API_KEY for Phase 2 (running agents autonomously in Python). Add it to .env when you get there.

Four skills are already installed and ready to run:

Run a built-in skill immediately

Before building your own skill, run one of the built-in examples so you understand what you're aiming for.

If your prototype involves any kind of research, scoring, or content generation, the lead qualifier is the best starting point. Run it now: 

/qualify-lead Retool

Claude will:

  2. Search for recent job postings (AE, SDR, revenue roles)

  4. Write a qualification report to output/leads/retool-qualification.md

Write your own skill

Now translate your Step 4 product brief into a skill. The fastest path is using the wizard: 

/new-skill

Claude will ask you three questions:

  1. What is the job? (one sentence)
  2. What are the inputs?
  3. What should it produce?

It creates .claude/skills/your-skill/SKILL.md for you. Here's what that file looks like for the lead qualifier:

---
name: qualify-lead
description: Qualify a company as a sales lead against our ICP.
  Use when asked to qualify, score, or evaluate a company.
---

# Lead Qualifier

## Steps
1. Search for the company: headcount, funding, business model, job postings
2. Score each ICP criterion: ✅ match / ⚠️ partial / ❌ mismatch / ❓ unknown
3. Assign tier: HOT, WARM, COLD, or DISQUALIFIED
4. Write report to output/leads/{company-slug}-qualification.md

## ICP
Load icp.md if it exists. Default: B2B SaaS, 50–500 employees, Series A–C.

## Output format
# Lead Qualification: {company}
Tier: {HOT|WARM|COLD|DISQUALIFIED}

## Decision
[3-5 sentences: business model, size, stage, signals]

## ICP Scorecard
| Criterion | Score | Evidence |
...

## Recommended next step
[Single concrete action with owner and channel]

## Sources
[URL for every factual claim]

The three skill patterns

Most AI features fit one of these shapes. Pick the one closest to your prototype:

PatternShapeUse for
Research & Synthesize Input → Search → Gather → Synthesize → Report Prospect research, market analysis, due diligence
Score & Classify Input → Load rubric → Analyze → Score → Decision Lead scoring, ticket routing, content moderation
Generate & Iterate Brief → Research → Draft → Variations → Output Copy, outreach, docs, content at scale

What makes a skill work

The most common reason a skill produces bad output: vague instructions. The fix is always to make the steps more specific.

Instead of thisWrite this
"Research the company" "Search for: company name + 'headcount', company name + 'funding', company name + 'jobs'"
"Write a report" "Write a markdown file with exactly these sections: Decision, Scorecard, Next Step, Sources"
"Use your judgment" "If headcount is unavailable, mark it ❓ and do NOT estimate"
"Make it useful" "Recommended next step: one action, one owner, one channel. Example: 'Email VP Sales at Retool to book a 30-min discovery call.'"

Run your skill on 3–5 real inputs from your actual use case. Each time the output isn't right, open SKILL.md, make the instructions more specific, and re-run. This is the prototyping loop.

Load a config file. If your skill needs criteria that will change over time (an ICP, brand voice guidelines, scoring rubrics), put them in a separate file (icp.md, brand-voice.md) and tell your skill to read it. This keeps your business logic editable without touching the skill itself.

Ship your MVP — export to production Python

Your skill works on a few examples. That's enough. Ship it now — before you write a single test case. 

/export-to-sdk

Claude generates sdk/your-skill/agent.py with the same instructions from your SKILL.md — same prompt, same tools, now running autonomously: 

import asyncio
from claude_agent_sdk import query, ClaudeAgentOptions
from pathlib import Path

PROMPT_TEMPLATE = """
Qualify this company as a sales lead: {company}

ICP criteria:
{icp}

Steps:
1. Search for headcount, funding, business model, and job postings
2. Score each criterion: ✅ match / ⚠️ partial / ❌ mismatch / ❓ unknown
3. Assign tier: HOT, WARM, COLD, or DISQUALIFIED
4. Write report to output/leads/{slug}-qualification.md
5. Return JSON: {{"company": "...", "tier": "...", "decision": "..."}}
"""

async def qualify(company: str) -> dict:
    icp = Path("icp.md").read_text() if Path("icp.md").exists() else ""
    result = None
    async for message in query(
        prompt=PROMPT_TEMPLATE.format(company=company, icp=icp, slug=company.lower()),
        options=ClaudeAgentOptions(
            allowed_tools=["WebSearch", "WebFetch", "Write"],
            permission_mode="acceptEdits",
        ),
    ):
        if hasattr(message, "result"):
            result = message.result
    if result is None:
        raise RuntimeError("Agent returned no result. Check ANTHROPIC_API_KEY.")
    return result

if __name__ == "__main__":
    import sys
    company = sys.argv[1] if len(sys.argv) > 1 else "Retool"
    print(asyncio.run(qualify(company)))
# Single run
python3 sdk/qualify-lead/agent.py "Retool"

# Batch from CSV
python3 sdk/qualify-lead/agent.py --batch leads.csv --output results.csv

# FastAPI endpoint
@app.post("/qualify")
async def qualify_endpoint(req: QualifyRequest):
    result = await qualify(req.company)
    return {"tier": result["tier"], "decision": result["decision"]}

Why ship before evals? Real-world inputs reveal failure modes that hand-written test cases never predict. Get the agent running on real data first. Write your test cases from the failure patterns you actually observe. And because the prompt lives in SKILL.md, any improvements you make later take effect in production immediately — no code changes, no redeployment.

Now harden it — add test cases

You've seen real outputs now. Write test cases grounded in what the agent actually does — not what you hoped it would do.

Create .claude/skills/your-skill/evals/test-cases.md. Each test case is an input plus a list of binary (yes/no) quality criteria:

## Test 1: Happy path — clear fit
**Input:** Retool

**Criteria:**
- [ ] Tier is HOT or WARM
- [ ] ICP Scorecard table is present with at least 4 rows
- [ ] Every scorecard row has ✅, ⚠️, ❌, or ❓ — no blank scores
- [ ] Recommended next step names a specific owner and channel
- [ ] At least 2 source URLs are cited

---

## Test 2: Clear disqualify
**Input:** Duolingo

**Criteria:**
- [ ] Tier is COLD or DISQUALIFIED
- [ ] Output mentions consumer focus or B2C as the reason
- [ ] Does NOT recommend booking a discovery call
- [ ] Sources are cited

---

## Test 3: Edge case — missing information
**Input:** "Runway Financial"

**Criteria:**
- [ ] Report is produced (not an error)
- [ ] Unknown data points are marked ❓, not guessed
- [ ] Does NOT fabricate headcount or funding without a source
- [ ] Recommended next step is still specific and actionable

What makes a good criterion

Good criteria are binary (yes or no), specific, and outcome-focused:

  • Good: "Does the output contain a source URL for each specific number cited?"
  • Good: "Is the tier exactly one of: HOT, WARM, COLD, DISQUALIFIED?"
  • Bad: "Is the writing good?" (subjective)
  • Bad: "Is the output comprehensive?" (undefined)

Aim for 10–20 test cases. Cover: the happy path, the expected failure (a clearly wrong input), edge cases (ambiguous inputs, missing data), and the failure patterns you've already seen in production.

Measure and improve

Run the eval: 

/eval-skill your-skill

Claude runs each test case, grades the outputs against your criteria, finds patterns in what failed, and writes a report to output/evals/your-skill-report.md. A typical first-run report: 

# Eval Report: qualify-lead
Run: 2026-06-14 | Score: 74% (17/23 criteria)

## Summary
74% — Needs improvement (target: 85%)

| Test case             | Score      | Status |
|-----------------------|------------|--------|
| Retool (hot lead)     | 100% (5/5) | ✅     |
| Duolingo (disqualify) | 80% (4/5)  | ⚠️     |
| Notion (edge case)    | 60% (3/5)  | ⚠️     |
| Runway (limited info) | 40% (2/5)  | ❌     |

## Top failure patterns

### 1. Missing source citations on low-info companies
Root cause: steps don't require citing a source for every data point.
Fix: add "cite the source URL for every specific number" to Step 2.

### 2. Fabricated headcount when data is unavailable
Root cause: no anti-hallucination instruction.
Fix: add "if headcount is unavailable, mark it ❓ — do NOT estimate."

Then auto-fix it: 

/improve-skill your-skill

Claude reads the failure patterns and edits SKILL.md directly to fix them. Because agent.py uses the same prompt template, the improvement takes effect in production immediately — no code changes, no redeployment. Re-run /eval-skill to measure. Repeat until 85%+.

See it in action: 65% → 100% in one iteration

The repo ships with a live demo of this loop. The qualify-lead-demo skill starts as a 22-line first draft that scores 65%. After one improve run it hits 100% — with version snapshots saved so you can see the exact diff. 

# Run the demo yourself
python3 examples/04-evals-and-improvement/sdk/eval_runner.py qualify-lead-demo
# → 65% (11/17 criteria)

python3 examples/04-evals-and-improvement/sdk/improve.py qualify-lead-demo --save-versions
# → edits SKILL.md: adds ICP, tier rubric, output format, anti-fabrication rule

python3 examples/04-evals-and-improvement/sdk/eval_runner.py qualify-lead-demo
# → 100% (17/17 criteria)

The --save-versions flag snapshots SKILL.md before and after each iteration to .claude/skills/qualify-lead-demo/versions/. In normal use you rely on git history — this flag is for demos and audits.

Automate the loop in the SDK

For skills that need many improvement cycles, or for running evals in CI, the SDK scripts automate the full loop: 

# Set up Python environment (one time)
python3 -m venv .venv && source .venv/bin/activate
python3 -m pip install claude-agent-sdk
source .env   # loads ANTHROPIC_API_KEY

# Run: eval → improve → eval → until target score
python3 examples/04-evals-and-improvement/sdk/improve.py your-skill \
  --target-score 0.85 \
  --max-iterations 5

This runs without supervision: evaluate, find failures, edit SKILL.md, re-evaluate, repeat — until the target score is hit or max iterations is reached. Run it overnight on a complex skill and come back to a passing eval.
