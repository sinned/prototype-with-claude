Evals & Improvement

How to know if your skill actually works

Running a skill on 3 examples isn't enough. Evals give you a systematic, reproducible measure of quality — and the improvement loop fixes failures automatically.

The problem with "it looks okay"

You run your skill. The output looks good. You ship it. Then in production, 30% of inputs produce garbage output.

The problem: LLM outputs vary. What works on your hand-picked test inputs might fail on real-world inputs. Traditional unit tests don't work ("assert output == expected") because outputs are never identical.

The solution: LLM-as-judge. Write binary quality criteria (yes/no, not "sort of"). Have Claude grade outputs against them. Find patterns in what fails. Fix the SKILL.md. Repeat.

1. Define Write test cases: an input + a list of yes/no quality criteria. What does "good output" actually mean?
2. Run Execute the skill on every test input. /eval-skill does this for you.
3. Grade Claude judges each output against the criteria. ✅ pass or ❌ fail — no partial credit.
4. Analyze Group failures by type. What's the root cause in the SKILL.md instructions?
5. Improve /improve-skill reads the failure patterns and edits your SKILL.md. Surgical changes only.

Writing good test cases

Test cases live in .claude/skills/<skill-name>/evals/test-cases.md. Each case has an input and a list of binary criteria.

## Test 1: Well-known company (happy path)
**Input:** Stripe

**Criteria:**
- [ ] Output is saved to output/stripe-brief.md
- [ ] Contains a TL;DR section
- [ ] Mentions payment processing as core business
- [ ] Includes at least one specific number (employees, revenue, or growth figure)
- [ ] Has at least 3 source URLs
- [ ] Recent developments section mentions something from the last 12 months

## Test 2: Ambiguous input
**Input:** Apple

**Criteria:**
- [ ] Defaults to Apple Inc. (the tech company), not apple the fruit
- [ ] Notes the assumption somewhere in the output
- [ ] Correctly describes their main business (hardware, software, services)
- [ ] Output is still complete and useful despite the ambiguity

## Test 3: Very little public information
**Input:** "Loom before it was acquired by Atlassian"

**Criteria:**
- [ ] Does NOT make up information it can't verify
- [ ] Acknowledges information constraints
- [ ] Still produces a useful brief with what's available

Good criteria are:

✓ Binary
Yes or no. "Does it include a source URL?" — not "is the sourcing adequate?"
✓ Specific
"Mentions company headcount as a number" — not "includes company info."
✓ Outcome-focused
"Would this help a sales rep prepare for a call?" — not "is it well-formatted?"
✗ Avoid
Criteria that require human judgment: "is the writing good?" or "is the tone appropriate?"

How many test cases? 10–20 per skill. Cover: happy path, edge cases, ambiguous inputs, and adversarial inputs (designed to break the skill). Add new cases whenever you find a production failure.

Running evals in Claude Code

  1. Create your test cases

    mkdir -p .claude/skills/my-skill/evals
cp examples/04-evals-and-improvement/test-cases-example.md \
   .claude/skills/my-skill/evals/test-cases.md
# Edit the file with your actual inputs and criteria

  2. Run the eval

    /eval-skill my-skill

    Claude runs each test case, grades the outputs, and writes a report to output/evals/my-skill-report.md.

  3. Review the report

    The report shows: overall score, per-test scores, failure patterns, and recommendations. If score ≥ 80%, you're in good shape. Below 80%: run improve.

  4. Improve the skill

    /improve-skill my-skill

    Claude edits your SKILL.md to fix the failure patterns. Then re-run /eval-skill to measure the improvement.

Automate the improvement loop

The eval loop runs as an Agent SDK script too — useful for CI/CD or when you want to run many improvement cycles without supervision. 

# Run evals once and get a report
python examples/04-evals-and-improvement/sdk/eval_runner.py my-skill

# Run the full loop: eval → improve → eval → improve → until target score
python examples/04-evals-and-improvement/sdk/improve.py my-skill \
  --target-score 0.85 \
  --max-iterations 5

The improvement loop:

  1. Runs your test suite and calculates a score
  2. If score ≥ target: stops and reports success
  3. Otherwise: asks Claude to analyze failures and edit the SKILL.md
  4. Re-runs evals to measure the change
  5. Appends a changelog to the report showing what changed and why
  6. Repeats until target score or max iterations

In CI/CD: Run the eval script on every PR that modifies a SKILL.md. Fail the build if score drops below threshold. This prevents prompt regressions from shipping.
