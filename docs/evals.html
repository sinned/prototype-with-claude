Evals & Improvement

Harden your agent after you ship

You shipped your first agent. Now use evals to systematically raise quality on the already-running production agent — no redeployment, no rewrite.

Evals are not a pre-ship gate

Most AI quality frameworks treat evals as something you do before shipping. That's backwards. Real-world inputs reveal failure modes that hand-written test cases never predict. Ship first. Get data. Then measure.

The problem evals solve: LLM outputs vary. What works on your hand-picked examples may fail on real inputs at scale. Traditional unit tests don't work ("assert output == expected") because outputs are never identical.

The solution: LLM-as-judge. Write binary quality criteria (yes/no, not "sort of"). Have Claude grade your production agent's outputs against them. Find patterns in what fails. Fix the SKILL.md. The improvement takes effect immediately — same prompt template, no redeployment.

Writing good test cases

Test cases live in .claude/skills/<skill-name>/evals/test-cases.md. Each case has an input and a list of binary criteria.

## Test 1: Well-known company (happy path)
**Input:** Stripe

**Criteria:**
- [ ] Output is saved to output/stripe-brief.md
- [ ] Contains a TL;DR section
- [ ] Mentions payment processing as core business
- [ ] Includes at least one specific number (employees, revenue, or growth figure)
- [ ] Has at least 3 source URLs
- [ ] Recent developments section mentions something from the last 12 months

## Test 2: Ambiguous input
**Input:** Apple

**Criteria:**
- [ ] Defaults to Apple Inc. (the tech company), not apple the fruit
- [ ] Notes the assumption somewhere in the output
- [ ] Correctly describes their main business (hardware, software, services)
- [ ] Output is still complete and useful despite the ambiguity

## Test 3: Very little public information
**Input:** "Loom before it was acquired by Atlassian"

**Criteria:**
- [ ] Does NOT make up information it can't verify
- [ ] Acknowledges information constraints
- [ ] Still produces a useful brief with what's available

Good criteria are:

✓ Binary
Yes or no. "Does it include a source URL?" — not "is the sourcing adequate?"
✓ Specific
"Mentions company headcount as a number" — not "includes company info."
✓ Outcome-focused
"Would this help a sales rep prepare for a call?" — not "is it well-formatted?"
✗ Avoid
Criteria that require human judgment: "is the writing good?" or "is the tone appropriate?"

How many test cases? 10–20 per skill. Cover: happy path, edge cases, ambiguous inputs, and adversarial inputs (designed to break the skill). Add new cases whenever you find a production failure.

Running evals in Claude Code

  1. Create your test cases

    mkdir -p .claude/skills/my-skill/evals
cp examples/04-evals-and-improvement/test-cases-example.md \
   .claude/skills/my-skill/evals/test-cases.md
# Edit the file with your actual inputs and criteria

  2. Run the eval

    /eval-skill my-skill

    Claude runs each test case, grades the outputs, and writes a report to output/evals/my-skill-report.md.

  3. Review the report

    The report shows: overall score, per-test scores, failure patterns, and recommendations. If score ≥ 80%, you're in good shape. Below 80%: run improve.

  4. Improve the skill

    /improve-skill my-skill

    Claude edits your SKILL.md to fix the failure patterns. Then re-run /eval-skill to measure the improvement.

Automate the improvement loop

The eval loop runs as an Agent SDK script too — useful for CI/CD or when you want to run many improvement cycles without supervision. 

# Run evals once and get a report
python3 examples/04-evals-and-improvement/sdk/eval_runner.py my-skill

# Run the full loop: eval → improve → eval → improve → until target score
python3 examples/04-evals-and-improvement/sdk/improve.py my-skill \
  --target-score 0.85 \
  --max-iterations 5

# Save snapshots of SKILL.md before/after each iteration (useful for demos and audits)
python3 examples/04-evals-and-improvement/sdk/improve.py my-skill \
  --target-score 0.85 \
  --save-versions

The improvement loop:

  1. Runs your test suite and calculates a score
  2. If score ≥ target: stops and reports success
  3. Otherwise: asks Claude to analyze failures and edit the SKILL.md
  4. Re-runs evals to measure the change
  5. Appends a changelog to the report showing what changed and why
  6. Repeats until target score or max iterations

The --save-versions flag is off by default — git history is the right place to track changes in normal use. Turn it on when you want a clean audit trail: snapshots land in .claude/skills/<name>/versions/ as SKILL-v0-initial.md, SKILL-v1-after-improve.md, etc.

In CI/CD: Run the eval script on every PR that modifies a SKILL.md. Fail the build if score drops below threshold. This prevents prompt regressions from shipping.

See it in action — 65% to 100% in one iteration

The repo ships with a real, runnable demo: a deliberately weak skill that starts at 65% and hits 100% after one improve.py call. Every artifact is checked in — you can see exactly what changed and why.

The starting point (22 lines, intentionally vague)

File: .claude/skills/qualify-lead-demo/versions/SKILL-v0-initial.md

---
name: qualify-lead-demo
description: A deliberately weak first-draft lead qualifier.
---

# Lead Qualifier (Demo — Iteration 0)

## Input
A company name.

## Steps
1. Look up the company.
2. Figure out if it's a good fit for our product.
3. Write something useful.

## Output
Save your assessment as a markdown file in `output/leads/`.

That's it — no ICP, no rubric, no output format, no error handling. A real "I'll fill this in later" first draft.

Top failures: no tier label (Claude writes "good fit" instead of HOT/WARM/COLD), no recommended next step, fabricated numbers without sources.

One command fixes it

# Reset the demo skill to v0 (already done in the repo)
cp .claude/skills/qualify-lead-demo/versions/SKILL-v0-initial.md \
   .claude/skills/qualify-lead-demo/SKILL.md

# Run the improvement loop, saving version snapshots
python3 examples/04-evals-and-improvement/sdk/improve.py qualify-lead-demo \
  --save-versions --max-iterations 1

# Re-evaluate
python3 examples/04-evals-and-improvement/sdk/eval_runner.py qualify-lead-demo

What the improve loop added: an explicit ICP definition, a tier rubric (HOT/WARM/COLD/DISQUALIFIED), a required output format with examples of good vs. bad next steps, and an anti-fabrication rule ("if you cannot find a data point, say so — do NOT fabricate"). 22 lines → 47 lines. Total API time: ~3 minutes.
